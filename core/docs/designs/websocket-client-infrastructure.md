## Vision

### 10x Check
The 10x version is a full collaborative workspace SDK: presence indicators showing who's viewing what, cursor tracking in block editors, real-time conflict resolution with operational transforms, offline queue with sync-on-reconnect, cross-tab coordination so multiple browser tabs share one WS connection. That's the 12-month vision. This plan builds the foundation layer that makes all of it possible.

**Explicitly NOT in scope:** presence, cursors, offline queue, cross-tab WS sharing, operational transforms.

## Prerequisites

**Backend `summary.userDisplayName` field:** Toast notifications depend on the backend including `userDisplayName` in the `WorkspaceEvent.summary` map. Verify this field is emitted by `EntityService`, `NotificationService`, etc. before implementing toasts. If absent, add it to the backend event publishers first — this is a small change (one line per publisher to include the actor's display name from `AuthTokenService`).

**Backend SockJS fallback:** Add `.withSockJS()` to the STOMP endpoint registration in `WebSocketConfig.kt`. Without this, the endpoint only accepts raw WebSocket — corporate proxies, VPNs, and some mobile networks silently kill raw WS. SockJS provides automatic fallback to HTTP long-polling. One-line backend change. Frontend: use `sockjs-client` as the WebSocket factory for `@stomp/stompjs`.

## Implementation Approach
**Approach B (revised): Invalidation-First with Targeted setQueryData**

The backend `summary` field is `Map<String, Any?>` with minimal data (entity type name, entity ID). It does NOT carry full entity objects. Therefore: entities use invalidation-only. `setQueryData` is reserved for simple cache updates where the summary carries enough data (e.g., notification read status).

The bridge layer routes each event through a strategy map keyed on `(channel, operation)`:

```typescript
type CacheStrategy = 'invalidate' | 'setQueryData';

const strategyMap: Record<WebSocketChannel, Record<OperationType, CacheStrategy>> = {
  ENTITIES: {
    CREATE: 'invalidate',    // Summary lacks full entity data
    UPDATE: 'invalidate',    // Summary lacks full entity data — can't safely setQueryData
    DELETE: 'invalidate',    // Summary lacks full entity data
  },
  NOTIFICATIONS: {
    CREATE: 'invalidate',    // Notification badge count + list need full refetch
    UPDATE: 'setQueryData',  // Mark as read (simple boolean flip, summary has enough)
    DELETE: 'invalidate',    // Badge count recalculation
  },
  BLOCKS: {
    CREATE: 'invalidate',    // Block tree structure changed
    UPDATE: 'invalidate',    // Block content/layout changed — complex nested structure
    DELETE: 'invalidate',    // Block tree structure changed
  },
  WORKFLOWS: {
    CREATE: 'invalidate',    // Workflow list changed
    UPDATE: 'invalidate',    // Workflow definition/execution state — complex nested structure
    DELETE: 'invalidate',    // Workflow list changed
  },
  WORKSPACE: {
    CREATE: 'invalidate',    // Membership/settings changed
    UPDATE: 'invalidate',    // Workspace properties changed
    DELETE: 'invalidate',    // Workspace-level deletion (rare)
  },
};
```

**Existing cache helpers (DO NOT recreate):** `updateEntityInPages()` and `removeEntitiesFromPages()` already exist at `apps/client/components/feature-modules/entity/hooks/mutation/instance/entity-cache.utils.ts`. These are used by entity mutation hooks. If future setQueryData paths are added (e.g., when backend summary carries full entity data), import from this existing location. `removeEntitiesFromPages()` takes a `Set<string>` — wrap single ID as `new Set([entityId])`.

**Fallback rule:** If `setQueryData` fails (item not found in cache, cache empty), fall back to `invalidateQueries`. setQueryData is an optimization, invalidation is the safety net.

**Future upgrade path:** When the backend enriches `summary` with full entity data, individual channels can be switched from `invalidate` to `setQueryData` by changing the strategy map entry. No architectural changes needed.

  - `/topic/workspace/{workspaceId}/blocks`
  - `/topic/workspace/{workspaceId}/workflows`
  - `/topic/workspace/{workspaceId}/notifications`

### WebSocketMessage DTO (backend shape to mirror)

```typescript
interface WebSocketMessage {
  channel: 'ENTITIES' | 'BLOCKS' | 'WORKFLOWS' | 'NOTIFICATIONS' | 'WORKSPACE';
  operation: 'CREATE' | 'UPDATE' | 'DELETE';  // Maps to backend OperationType enum
  workspaceId: string;  // UUID
  entityId: string | null;  // UUID, null for workspace-level events

```

Hand-written (not OpenAPI-generated) since WebSocket messages don't flow through REST endpoints.

## Key Architectural Decisions

### Library: @stomp/stompjs
Direct usage, no wrapper library like `react-stomp-hooks`. The wrapper adds a dependency for something achievable with a single provider + hook pattern matching the codebase's existing Zustand factory conventions.

### Provider placement
```
ThemeProvider
  └── AuthProvider
      └── QueryClientWrapper
          └── WebSocketProvider   ← NEW (needs both session and queryClient)
              └── StoreProviderWrapper
                  └── <main>
```

### SSR safety
`@stomp/stompjs` accesses `WebSocket` at import time. The provider MUST:
- Be a `'use client'` component
- Use dynamic `import()` inside `useEffect` to load `@stomp/stompjs` — `typeof window` guards alone are insufficient when the module accesses globals at import time
- Example: `useEffect(() => { import('@stomp/stompjs').then(({ Client }) => { ... }) }, [])`
- Alternative: `next/dynamic` with `ssr: false` for the provider component itself

### Token refresh strategy
Backend validates JWT on CONNECT frame only. Supabase JWTs expire every ~60 minutes. Strategy:
1. Listen to Supabase `onAuthStateChange` for `TOKEN_REFRESHED` events
2. On token refresh: deactivate STOMP client, reactivate with new token
3. Reconnection triggers the catch-up invalidation (expansion #3), closing any gap
4. **Known risk:** Between token expiry and Supabase auto-refresh (~seconds), WS may be stale. Acceptable for v1.

### Self-event suppression
Compare `message.userId` against `useAuth().user.id`. If match, skip cache updates.

**Important nuance:** Self-suppression assumes the triggering mutation already performed an optimistic update or invalidation via its `onSuccess` callback. This is true for entity mutations (which use `setQueriesData` in `onSuccess`) but may not hold for all mutation hooks. **Rule: only suppress for channels/mutations known to have client-side cache updates.** For mutations without optimistic updates, the WS event is the only update path — do NOT suppress. The bridge should check `suppressSelf` per strategy entry, defaulting to `false` (safe default).

Updated strategy type:
```typescript
type CacheStrategyEntry = {
  action: 'invalidate' | 'setQueryData' | 'both';
  suppressSelf: boolean;  // true = skip if userId matches current user
};
```

### Reconnection behavior
`@stomp/stompjs` has built-in reconnect with configurable delay (default: 5s). On successful reconnect:
1. Re-subscribe to workspace topics (automatic with stompjs `activate()`)
2. Invalidate ALL queries matching these prefixes for the active workspace, using `refetchType: 'active'` to only refetch currently-mounted queries (avoids refetching scrolled-past pages or navigated-away views):
   - `queryClient.invalidateQueries({ queryKey: ['entities', workspaceId], refetchType: 'active' })`
   - `queryClient.invalidateQueries({ queryKey: ['blocks', workspaceId], refetchType: 'active' })`
   - `queryClient.invalidateQueries({ queryKey: ['workflows', workspaceId], refetchType: 'active' })`
   - `queryClient.invalidateQueries({ queryKey: ['notifications', workspaceId], refetchType: 'active' })`
   - `queryClient.invalidateQueries({ queryKey: ['workspace', workspaceId], refetchType: 'active' })`
3. This is a coarser fallback than the per-event strategy — intentional. During disconnect window, events were lost. Invalidation ensures catch-up. `refetchType: 'active'` scopes the storm to only what the user is looking at.

### Tab visibility handler
On `document.visibilitychange` to `'visible'`:
1. Check if STOMP client is still connected (heartbeats may have stalled while backgrounded)
2. If stale or disconnected: force deactivate + reactivate (triggers reconnect + catch-up)
3. If still connected: fire catch-up invalidation anyway (events may have been missed while backgrounded, even if connection survived)
4. Implemented in `WebSocketProvider` via `useEffect` with `visibilitychange` event listener + cleanup

### Auth failure handling
When the server rejects the CONNECT frame (invalid JWT, revoked workspace role):
- STOMP client receives an ERROR frame with a message
- Connection status transitions to **Red** (disconnected)
- If the error message indicates auth failure (e.g., "401" or "Authentication failed"):
  - Do NOT auto-reconnect (reconnecting with the same bad token is pointless)
  - Show a toast: "Real-time connection lost. Please refresh the page."
  - If the auth session itself is invalid (Supabase session expired), the existing `AuthGuard` will redirect to login on the next API call
- If the error is not auth-related (server crash, network): allow normal reconnect cycle

### Operations strategy
- **CREATE:** Always invalidate. Cannot reliably insert into `InfiniteData` (don't know which page, sort position, or filter match).
- **UPDATE:** `setQueryData` via `updateEntityInPages()` helper for list queries. Also update detail query cache if it exists. Fall back to invalidate if item not found.
- **DELETE:** `setQueryData` via `removeEntityFromPages()` (new helper, ~15 lines) to remove item from `InfiniteData` pages. Invalidate if item not found.

## Connection Status Indicator

- **Location:** App shell navbar, next to the workspace name/selector
- **States:**
  - Green dot: STOMP connected, receiving heartbeats
  - Yellow dot (pulse animation): Reconnecting (STOMP client in reconnect cycle)
  - Red dot: Disconnected (failed after max reconnect attempts or auth error)
  - No dot: WS disabled (e.g., not authenticated yet)
- **Interaction:** Hover tooltip showing state + last connected timestamp
- **Not clickable** in v1

## Toast on Remote Changes

Uses existing `sonner` toast system. Triggered by the query bridge when `message.userId !== currentUserId`.

- Format: `"{displayName} {operation} {entityTypeName}: {entityDisplayName}"` (e.g., "Jared updated Contact: Acme Corp")
- Actor display name comes from `message.summary.userDisplayName` — see Prerequisites section for backend dependency
- **Rate limiting:** Max 3 toasts per 5 seconds. After that, batch into "N more changes" summary toast. Prevents toast flooding during bulk operations.
- **Channels with toasts:** ENTITIES (create/update/delete), NOTIFICATIONS (create only). BLOCKS, WORKFLOWS, WORKSPACE: no toasts (too noisy or not user-relevant).

## Dev-Mode Logging

- Console logging of all WS messages with structured formatting: `[WS] {channel} {operation} {entityId} @ {timestamp}`
- Gated behind `process.env.NODE_ENV === 'development'` — production bundles must NOT include WS console logging
- Located in `lib/websocket/dev-logger.ts`, called by `use-workspace-subscription`

## Known Risks

1. **Token expiry window** — Up to ~seconds of stale WS between JWT expiry and Supabase auto-refresh. Acceptable for v1. Mitigated by reconnect catch-up invalidation.
2. **Token refresh triggers full invalidation** — Reconnect-on-token-refresh fires a full cache invalidation every ~60 minutes per session. For workspaces with large entity lists, this is a predictable refetch storm. Acceptable for v1 at current user count. Future improvement: STOMP header-based token update without reconnect (requires backend changes).
3. **InfiniteData setQueryData fragility** — If TanStack Query's internal page structure changes between versions, the `updateEntityInPages` helper could break. Mitigated by falling back to invalidation.
4. **CORS for WS in local dev** — Backend at `localhost:8081`, frontend at `localhost:3001`. The `WebSocketConfig` must include dev origins in `allowedOrigins`. Already handled by existing `SecurityConfig` CORS setup — verify it covers WS upgrade requests.
5. **Multi-tab behavior** — Each tab opens its own STOMP connection. For a workspace with 5 users and 3 tabs each, that's 15 connections. Acceptable for current scale. Cross-tab WS sharing deferred to 12-month vision.
