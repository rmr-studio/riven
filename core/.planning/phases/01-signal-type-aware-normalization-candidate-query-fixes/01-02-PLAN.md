    Make the following changes to `IdentityMatchCandidateService.kt`:

    **1. Constructor injection (add IdentityNormalizationService):**
    Add `private val normalizationService: IdentityNormalizationService` to the constructor alongside existing `entityManager` and `logger`.

    **3. Bump CANDIDATE_LIMIT from 50 to 100** (per locked decision — more rows flowing without DISTINCT ON collapse).

    **4. Remove DISTINCT ON from runCandidateQuery SQL:**
    Change the SQL from:
    ```sql
    SELECT DISTINCT ON (ea.entity_id)
        ea.entity_id AS candidate_entity_id, ...
    ORDER BY ea.entity_id, sim_score DESC
    ```
    To (remove the subquery wrapper entirely since DISTINCT ON was the only reason for it):
    ```sql
    SELECT ea.entity_id AS candidate_entity_id,
           ea.attribute_id AS candidate_attribute_id,
           ea.value->>'value' AS candidate_value,
           similarity(ea.value->>'value', :inputValue) AS sim_score
    FROM entity_attributes ea
    JOIN entity_type_semantic_metadata sm
        ON sm.workspace_id = :workspaceId
       AND sm.target_type = 'ATTRIBUTE'
       AND sm.target_id = ea.attribute_id
       AND sm.classification = 'IDENTIFIER'
       AND sm.deleted = false
    WHERE ea.workspace_id = :workspaceId
      AND ea.entity_id != :triggerEntityId
      AND ea.deleted = false
      AND (ea.value->>'value') % :inputValue
      AND similarity(ea.value->>'value', :inputValue) > 0.3
    ORDER BY sim_score DESC
    LIMIT $CANDIDATE_LIMIT
    ```

    **5. Change mergeCandidates groupBy key:**
    From: `.groupBy { it.candidateEntityId to it.signalType }`
    To: `.groupBy { it.candidateEntityId to it.candidateAttributeId }`
    This preserves multi-attribute rows while deduplicating trigram vs exact-digits for the same (entity, attribute) pair.

    **6. Fix requireNotNull (CAND-03):**
    From: `group.maxByOrNull { it.similarityScore }!!`
    To: `requireNotNull(group.maxByOrNull { it.similarityScore }) { "Candidate group was empty - groupBy should never produce an empty group" }`

    **7. Add findPhoneExactDigitsCandidates() private method:**
    New private method with native SQL query:
    ```sql
    SELECT ea.entity_id   AS candidate_entity_id,
           ea.attribute_id AS candidate_attribute_id,
           ea.value->>'value' AS candidate_value,
           1.0            AS sim_score
    FROM entity_attributes ea
    JOIN entity_type_semantic_metadata sm
        ON sm.workspace_id = :workspaceId
       AND sm.target_type = 'ATTRIBUTE'
       AND sm.target_id = ea.attribute_id
       AND sm.classification = 'IDENTIFIER'
       AND sm.deleted = false
    WHERE ea.workspace_id = :workspaceId
      AND ea.entity_id != :triggerEntityId
      AND ea.deleted = false
      AND regexp_replace(ea.value->>'value', '[^0-9]', '', 'g') = :normalizedDigits
    LIMIT $CANDIDATE_LIMIT
    ```
    Method signature: `private fun findPhoneExactDigitsCandidates(triggerEntityId: UUID, workspaceId: UUID, normalizedDigits: String): List<CandidateMatch>`
    Map results the same way as runCandidateQuery, using `MatchSignalType.PHONE` as signalType. sim_score is 1.0 (exact match).

    **8. Call findPhoneExactDigitsCandidates in findCandidates() loop:**
    After `allCandidates.addAll(candidates)`, add:
    ```kotlin
    if (signalType == MatchSignalType.PHONE) {
        val exactCandidates = findPhoneExactDigitsCandidates(triggerEntityId, workspaceId, normalizedValue)
        allCandidates.addAll(exactCandidates)
    }
    ```
    Union happens in Kotlin (per locked decision), NOT as SQL UNION. mergeCandidates handles dedup — keeps highest sim_score per (entityId, attributeId).

    **9. Update KDoc:**
    - Update `findCandidates()` KDoc to mention signal-type-aware normalization and phone exact-digits union

    - Add a KDoc comment on CANDIDATE_LIMIT noting it applies per-query, not globally (per pitfall 3)
