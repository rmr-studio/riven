  <action>
    Create `IntegrationSyncActivitiesImpl` as a Spring `@Service` bean implementing `IntegrationSyncActivities`.

    **Constructor dependencies (all via constructor injection):**
    - `IntegrationConnectionRepository` — connection lookup and status update
    - `IntegrationSyncStateRepository` — sync state read/write
    - `NangoClientWrapper` — paginated record fetching
    - `SchemaMappingService` — field mapping
    - `EntityRepository` — batch dedup query and entity save
    - `EntityAttributeService` — attribute persistence (no @PreAuthorize, safe to call from activity)
    - `EntityRelationshipService` — Pass 2 relationship linking
    - `IntegrationDefinitionRepository` — resolve integration slug for manifest lookup
    - `ManifestCatalogRepository` — resolve manifest for field mapping lookup
    - `CatalogFieldMappingRepository` — load field mappings for a model
    - `EntityTypeRepository` — resolve workspace entity type for this model
    - `KLogger` — logging

    **Activity 1: transitionToSyncing(connectionId, workspaceId)**
    - Find connection by ID. If not found, log warning and return (do not throw — workflow will handle missing connection via sync state FAILED)
    - If `connection.status == SYNCING`, log info and return (already there from another model's workflow)
    - If `!connection.status.canTransitionTo(SYNCING)`, log warning and return (do not throw)
    - Otherwise set `connection.status = SYNCING` and save
    - Keep this method short (~15 lines)

    **Activity 2: fetchAndProcessRecords(input) -> SyncProcessingResult**
    This is the long-running activity. Split into clear private methods.

    **Step 2a: Resolve model context**
    - Look up IntegrationDefinitionEntity by `input.integrationId`
    - Look up ManifestCatalogEntity by definition slug via `manifestCatalogRepository.findByKey(slug)`
    - Look up CatalogFieldMappingEntity by manifest ID + model via `catalogFieldMappingRepository.findByManifestIdAndEntityTypeKey(manifestId, input.model)`
    - Look up EntityType by matching key == model in workspace entity types for this integration. Use whatever query is available on EntityTypeRepository (check the actual repository; may need `findByWorkspaceId()` filtered by integration, or a more specific query). The entity type's `key` field should match `input.model`.
    - Extract fieldMappings (Map<String, ResolvedFieldMapping>) and keyMapping (Map<String, UUID>) from the CatalogFieldMappingEntity — check how existing code resolves these from the catalog field mapping. The mapping entity likely has a `mappings` JSONB field.
    - If any lookup fails, return `SyncProcessingResult(entityTypeId=UUID(0,0), cursor=null, recordsSynced=0, recordsFailed=0, lastErrorMessage="...", success=false)` — the finalize activity will write this as FAILED.

    **Step 2b: Determine cursor**
    - Check existing sync state: `syncStateRepository.findByIntegrationConnectionIdAndEntityTypeId(input.connectionId, entityTypeId)`
    - Use persisted `lastCursor` if it exists (source of truth), else fall back to `input.modifiedAfter` from webhook payload
    - This ensures the workflow re-fetches from the last successful position, not the webhook's modifiedAfter

    **Step 2c: Paginated fetch + process loop**
    ```
    var cursor = resolvedCursor
    val relationshipPending = mutableListOf<RelationshipPending>()
    var recordsSynced = 0
    var recordsFailed = 0
    var lastErrorMessage: String? = null

    do {
        val page = nangoClientWrapper.fetchRecords(providerConfigKey, nangoConnectionId, model, cursor, modifiedAfter)

        val (synced, failed, error) = processBatch(page.records, workspaceId, integrationId, entityTypeId, typeKey, identifierKey, fieldMappings, keyMapping, relationshipPending)
        recordsSynced += synced
        recordsFailed += failed
        if (error != null) lastErrorMessage = error

        cursor = page.nextCursor

        // Heartbeat after each page — stores cursor for retry resume
        Activity.getExecutionContext().heartbeat(cursor)
    } while (cursor != null)
    ```

    **Step 2d: processBatch (private method)**
    - Extract external IDs from records: `record.payload["id"] as? String` for each record
    - Batch dedup: `entityRepository.findByWorkspaceIdAndSourceIntegrationIdAndSourceExternalIdIn(workspaceId, integrationId, externalIds).associateBy { requireNotNull(it.sourceExternalId) }`
    - For each record in the batch, wrap in try-catch (SYNC-06):
      - Extract externalId from `record.payload["id"]`. If null, log error and count as failed, continue
      - Switch on `record.nangoMetadata.lastAction`:
        - **DELETED**: call handleDelete(existing[externalId]) — if null, no-op. If found, set `deleted=true`, `deletedAt=ZonedDateTime.now()`, save entity. Count as synced.
        - **ADDED** or **UPDATED**: call mapPayload with record.payload, fieldMappings, keyMapping. If `mappingResult.errors.isNotEmpty()`, log warning with error details, count as failed, continue. If warnings exist, log at debug level and proceed. Then call upsertEntity: if `existing[externalId] != null` → update (set lastSyncedAt, save entity, saveAttributes). If null → create new EntityEntity with sourceType=INTEGRATION, sourceIntegrationId, sourceExternalId, firstSyncedAt/lastSyncedAt=now. Save entity, saveAttributes. Count as synced.
        - Collect relationship data if the mapping result or record payload contains relationship fields — extract target external IDs and relationship definition key, add to relationshipPending list.
      - On any exception: log error with record external ID, increment failed counter, store exception message as lastErrorMessage, continue to next record

    **Step 2e: resolveRelationships (private method) — Pass 2**
    - Group pending relationships by relationshipDefinitionKey
    - For each group: batch lookup target entities by external ID via `entityRepository.findByWorkspaceIdAndSourceIntegrationIdAndSourceExternalIdIn()`
    - For each item in the group: resolve target entity IDs from the lookup map. Missing targets are skipped silently (best-effort, eventually consistent)
    - Use `EntityRelationshipService` to create the relationship links for resolved targets. NOTE: Check if EntityRelationshipService methods have @PreAuthorize. If they do, use the repository layer directly for relationship persistence. The activity has no JWT context.
    - Wrap each relationship resolution in try-catch — relationship failures should not fail the sync

    **Step 2f: Return SyncProcessingResult**

    **Activity 3: finalizeSyncState(connectionId, entityTypeId, result)**
    - Look up existing: `syncStateRepository.findByIntegrationConnectionIdAndEntityTypeId(connectionId, entityTypeId)`
    - If not found, create new `IntegrationSyncStateEntity(integrationConnectionId=connectionId, entityTypeId=entityTypeId)`
    - Set `status = if (result.success) SyncStatus.SUCCESS else SyncStatus.FAILED`
    - Set `lastRecordsSynced = result.recordsSynced`, `lastRecordsFailed = result.recordsFailed`, `lastErrorMessage = result.lastErrorMessage`
    - If success: set `lastCursor = result.cursor`, set `consecutiveFailureCount = 0`
    - If failure: do NOT update lastCursor, set `consecutiveFailureCount = existing.consecutiveFailureCount + 1` (or 1 if new)
    - Save entity

    **Update TemporalWorkerConfiguration:**
    - Now that `IntegrationSyncActivitiesImpl` exists, change the constructor parameter from nullable to non-nullable: `private val integrationSyncActivities: IntegrationSyncActivities`
    - Remove the conditional registration — register directly: `syncWorker.registerActivitiesImplementations(integrationSyncActivities)`

    **CRITICAL PITFALLS TO AVOID:**
    - Do NOT call `EntityService.saveEntity()` — it requires JWT context. Use `entityRepository.save()` + `entityAttributeService.saveAttributes()` directly.
    - Do NOT use `@PreAuthorize` on activity methods — Temporal calls them, not Spring AOP.
    - Do NOT store state in instance fields of the activity — it's a singleton. All data via method parameters.
    - DO call `Activity.getExecutionContext().heartbeat(cursor)` after every page.
    - Do NOT use `Workflow.sideEffect()` or `Workflow.getLogger()` inside activity code — those are workflow-only APIs.
