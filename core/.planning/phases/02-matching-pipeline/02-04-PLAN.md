    **IdentityMatchActivities.kt** (`riven.core.service.workflow.identity`):
    ```kotlin
    @ActivityInterface
    interface IdentityMatchActivities {
        @ActivityMethod
        fun findCandidates(entityId: UUID, workspaceId: UUID): List<CandidateMatch>

        @ActivityMethod
        fun scoreCandidates(triggerEntityId: UUID, workspaceId: UUID, candidates: List<CandidateMatch>): List<ScoredCandidate>

        @ActivityMethod
        fun persistSuggestions(workspaceId: UUID, scoredCandidates: List<ScoredCandidate>, userId: UUID?): Int
    }
    ```

    **IdentityMatchActivitiesImpl.kt** — Spring `@Component` bean:
    - Constructor inject: `IdentityMatchCandidateService`, `IdentityMatchScoringService`, `IdentityMatchSuggestionService`, `KLogger`
    - `findCandidates`: delegates to `candidateService.findCandidates(entityId, workspaceId)`
    - `scoreCandidates`: calls `candidateService.getTriggerAttributes(triggerEntityId, workspaceId)` to get the trigger attribute map, then delegates to `scoringService.scoreCandidates(triggerEntityId, triggerAttributes, candidates)`.
    - `persistSuggestions`: delegates to `suggestionService.persistSuggestions(workspaceId, scoredCandidates, userId)`

    **IdentityMatchWorkflow.kt** (`riven.core.service.workflow.identity`):
    ```kotlin
    @WorkflowInterface
    interface IdentityMatchWorkflow {
        @WorkflowMethod
        fun matchEntity(entityId: UUID, workspaceId: UUID, userId: UUID?): Int
    }
    ```

    **IdentityMatchWorkflowImpl.kt:**
    - NOT a Spring bean — Temporal manages lifecycle.
    - Create activity stub with `ActivityOptions`: startToCloseTimeout = 30s, retry policy = 3 attempts with exponential backoff.
    - `matchEntity` implementation:
      1. `val candidates = activities.findCandidates(entityId, workspaceId)` — if empty, return 0.
      2. `val scored = activities.scoreCandidates(entityId, workspaceId, candidates)` — if empty, return 0.
      3. `val count = activities.persistSuggestions(workspaceId, scored, userId)` — return count.

    **CRITICAL — Workflow ID for Temporal-level dedup (per CONTEXT.md locked decision):**
    When starting this workflow from a client (Phase 3 dispatcher or any caller), the caller MUST set `WorkflowOptions.Builder().setWorkflowId("identity-match-$entityId")` to ensure Temporal-level dedup. This is NOT set inside the workflow impl itself — it is a caller responsibility. However, to make this convention explicit and discoverable:
    1. Add a companion object to `IdentityMatchWorkflow` interface: `companion object { fun workflowId(entityId: UUID): String = "identity-match-$entityId" }` — callers use this to derive the correct workflow ID.
    2. Add a KDoc comment on the interface documenting the workflow ID convention: "Callers must set WorkflowOptions.setWorkflowId(IdentityMatchWorkflow.workflowId(entityId)) to ensure Temporal-level dedup for duplicate entity triggers."

    **TemporalWorkerConfiguration.kt changes:**
    - Add companion constant: `const val IDENTITY_MATCH_QUEUE = "identity.match"`
    - Inject `IdentityMatchActivitiesImpl` as constructor parameter.
    - In `workerFactory()`, after existing worker setup, add:
      ```kotlin
      val identityWorker = factory.newWorker(IDENTITY_MATCH_QUEUE)
      identityWorker.registerWorkflowImplementationFactory(IdentityMatchWorkflow::class.java) {
          IdentityMatchWorkflowImpl()
      }
      identityWorker.registerActivitiesImplementations(identityMatchActivities)
      ```
    - Log registration of identity workflow and activities.

    **IdentityMatchPipelineIntegrationTest.kt:**
    An integration test that validates the full matching pipeline by calling the three services in sequence against a real PostgreSQL instance. This does NOT test Temporal workflow execution (that requires a Temporal test server) — it tests the services end-to-end.

    Setup pattern: Follow `IdentityInfrastructureIntegrationTest` pattern:
    - `@SpringBootTest` with `@ActiveProfiles("integration")`
    - Testcontainers PostgreSQL with `@DynamicPropertySource`
    - `@BeforeAll` installs pg_trgm extension, identity constraints, and seeds test data via `jdbcTemplate`
    - Exclude Temporal and security auto-configuration (same exclusions as existing integration test)
    - `@EntityScan` scoped to `riven.core.entity.identity` and `riven.core.entity.entity` (need entity_attributes entities)

    Test data setup in `@BeforeAll`:
    1. Create two entity_type rows (e.g., "Contact" and "Lead") in the same workspace
    2. Create entity_type_semantic_metadata rows marking certain attributes as IDENTIFIER
    3. Create entity_attributes for Entity A: email="john@example.com", phone="555-1234"
    4. Create entity_attributes for Entity B: email="john@exmple.com" (typo — similar), phone="555-1234" (exact)
    5. Create entity_attributes for Entity C: email="totally@different.com" (no match)
    6. Create entity_attributes for Entity D in a DIFFERENT workspace (must never appear as candidate)

    Test cases:
    1. **Matching pair produces suggestion:** Call candidateService.findCandidates(entityA) -> scoringService.scoreCandidates(entityA, ..., candidates) -> suggestionService.persistSuggestions(...). Assert: one suggestion created for Entity A + Entity B with PENDING status, composite score > 0.5, signals JSONB contains EMAIL and PHONE entries.
    2. **No candidates above threshold:** Call pipeline for Entity C. Assert: no suggestion created.
    3. **Cross-workspace isolation:** Entity D from different workspace never appears in Entity A's candidates.
    4. **Idempotent re-run:** Call the full pipeline for Entity A again. Assert: still exactly one suggestion (duplicate silently skipped), return count = 0.
    5. **Re-suggestion after rejection:** Manually set the suggestion to REJECTED (soft-delete it per re-suggestion pattern), update entity B's email to "john@example.com" (exact match — higher score). Re-run pipeline. Assert: old rejected suggestion stays soft-deleted, new PENDING suggestion created with higher confidence score.

    Note: Since these are integration tests hitting real PostgreSQL, they will use `@Transactional` for test isolation or `@BeforeEach` cleanup. Follow the same pattern as `IdentityInfrastructureIntegrationTest`.

    The integration test proves all six Phase 2 success criteria from the ROADMAP.
