## Decisions Made

1. **Task 1 kept minimal per CONTEXT.md.** Plan 01 had already executed the Boot-3→Boot-4 test API sweep as part of its Rule 3 auto-fixes (8 test files rewritten for split autoconfig modules, JUnit-Jupiter-6 ExtensionContext non-null). No opportunistic clean-up of `Mockito.when` style (Known Inconsistency #4) — out of scope.
2. **Vector test targets `EntityEmbeddingEntity`, not `NoteEntity`** as the plan text suggested. `NoteEntity` does not carry a vector column — `pgvector` lives on `EntityEmbeddingEntity` via `@JdbcTypeCode(SqlTypes.VECTOR) + @Array(length=1536)` on a `FloatArray` field. Plan text was fixed via Rule 1 auto-fix (plan reference bug).
3. **JDBC parent-row seeding** chosen over going through `EntityTypeService` so the vector canary is not blocked by the pre-existing `SourceTypeJpaRoundTripTest` Hibernate 7 `NonSerializableObjectException` on `EntityTypeEntity.schema` JSONB — that belongs to Plan 04's closure gate, not this test's scope.
4. **Resilience4j test uses synthetic annotated bean.** Production code has zero `@CircuitBreaker` / `@Retry` annotation usage (only programmatic `CircuitBreaker.executeRunnable{}`). A real-target test is not possible; the wiring test is future-proofing per plan action step 3.
5. **`@Import(CircuitBreakerConfiguration, FallbackConfiguration, SpelResolverConfiguration)` is the canonical Boot-4-without-starter pattern** for direct-wired Resilience4j. These three cooperatively provide: registry + aspect + fallback executor + SpEL name resolution. `CompositeCustomizer` is created internally by `CircuitBreakerConfiguration` — do NOT redeclare it in the test config (triggers `BeanDefinitionOverrideException`).

### Auto-fixed Issues

**1. [Rule 1 - Bug] Vector test file path pointed at non-vector entity**
- **Found during:** Task 2 code-context gathering.
- **Issue:** Plan specified `core/src/test/kotlin/riven/core/entity/note/NoteEmbeddingHibernate7IntegrationTest.kt`, but `NoteEntity` carries no vector column. `pgvector` mapping lives on `EntityEmbeddingEntity`.
- **Fix:** Test created at `core/src/test/kotlin/riven/core/entity/enrichment/EntityEmbeddingHibernate7IntegrationTest.kt` against the actual pgvector-carrying entity. Plan intent (exercise H7 vector round-trip) fully satisfied.
- **Verification:** Both tests pass against Testcontainers pgvector.
- **Committed in:** `5cf3e4728`

**2. [Rule 1 - Bug] AssertJ `containsExactly(vararg Float)` cannot consume a `FloatArray` spread**
- **Found during:** Task 2 first compile.
- **Issue:** `assertThat(floatArray).containsExactly(*vector.toTypedArray())` — the `Array<Float>` spread expands to `Float` args, but AssertJ's `FloatArrayAssert.containsExactly` takes a `float[]` (primitive) overload that Kotlin's spread doesn't bind to cleanly. Compile error.
- **Fix:** Switched to `assertThat(reloaded.embedding).isEqualTo(vector)` — direct `FloatArray` equality.
- **Verification:** Compiles and passes.
- **Committed in:** `5cf3e4728`

**3. [Rule 1 - Bug] `CompositeCustomizer` bean redeclared in test config**
- **Found during:** Task 3 first Spring context load.
- **Issue:** Test config had a `@Bean fun compositeCircuitBreakerCustomizer()` but `CircuitBreakerConfiguration` (imported) already provides this same-named bean → `BeanDefinitionOverrideException`.
- **Fix:** Removed the duplicate `@Bean` method; `CircuitBreakerConfiguration.compositeCircuitBreakerCustomizer(...)` wires it automatically from an `emptyList()` of customizers.
- **Verification:** Both Resilience4j tests pass.
- **Committed in:** `940569ac8`

**4. [Rule 1 - Bug] Wrong import path for `SpelResolverConfiguration`**
- **Found during:** Task 3 first compile.
- **Issue:** Imported as `io.github.resilience4j.spelresolver.SpelResolverConfiguration`; the class actually lives under `.spelresolver.configure.SpelResolverConfiguration`.
- **Fix:** Corrected import path.
- **Verification:** Compiles.
- **Committed in:** `940569ac8`

**5. [Rule 1 - Bug] `InstanceProperties` field-access syntax assumed direct field access**
- **Found during:** Task 3 second compile.
- **Issue:** `CommonCircuitBreakerConfigurationProperties.InstanceProperties` is a Java class with private fields and setter-returning-this builders. Used `.apply { failureRateThreshold = 50.0f }` which tries field-access.
- **Fix:** Switched to setter-chain style: `.setFailureRateThreshold(50.0f).setSlidingWindowSize(4).set...`.
- **Verification:** Compiles and passes.
- **Committed in:** `940569ac8`
