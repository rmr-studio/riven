# Phase 03.1: Boot 3.5.3 → 4.0.x Upgrade with Transitive Dep Bumps - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Upgrade Spring Boot from 3.5.3 to 4.0.x and bump the transitive dependencies that break or refuse to coexist with Boot 4. Outcome: a green build/test suite and a manually-smoked Nango integration sync on Boot 4, so that Phase 03-04 (NL-assisted mapping) can pull Spring AI 2.x against a Boot 4 runtime.

Out of scope for this phase: any new feature work, the Spring AI integration itself (lives in 03-04), Java/Kotlin major bumps, opportunistic non-Spring dep refreshes.

</domain>

<decisions>
## Implementation Decisions

### Bump scope
- **Minimum-necessary** policy: bump only deps that fail to compile or run on Boot 4. Anything that still works on Boot 3.5 is left pinned.
- **Spring family unpins**: remove the version pins on `spring-boot-starter-oauth2-resource-server:3.5.0` and `spring-security-oauth2-jose:6.5.0` so the Boot 4 BOM manages versions.
- **SpringDoc**: bump `springdoc-openapi-starter-webmvc-ui` from 2.8.6 to the 3.x line that targets Boot 4.
- **Hibernate**: bump to the Hibernate 7 line. Replace `hypersistence-utils-hibernate-63:3.9.2` with the `hibernate-70` (or current H7) artifact and bump `hibernate-vector:6.6.18.Final` to the matching 7.x release. pgvector mapping changes are expected and must be covered by tests.
- **Java/Kotlin stay pinned**: Java 21 + Kotlin 2.1.21. JVM/Kotlin major bumps are deferred to their own future phase (one risk axis at a time).

### Laggard deps
- **Temporal starter**: a Boot 4 release exists — bump `temporal-spring-boot-starter` to it (no shim needed). Keep `temporal-sdk` and `temporal-kotlin` aligned.
- **Resilience4j**: do not wait for an official `resilience4j-spring-boot4` starter. Drop `resilience4j-spring-boot3:2.3.0` and wire `resilience4j-core` + `-aop` directly via Spring AOP config.
- **Other low-risk deps** (ShedLock, Bucket4j, etc.): bump aggressively to whatever Boot-4-aware version exists. If a dep is genuinely stuck, swap library or defer the feature it backs.
- **Stop condition**: if 2+ critical deps are blocking, pause Phase 03.1 and re-defer Phase 03-04 further. Do not merge a half-upgraded codebase.

### Deprecation cleanup
- **Compile-clean + runtime-critical only** for general deprecations. Non-critical deprecation warnings are logged as tech-debt items, not fixed in this phase.
- **Spring Security 7**: migrate `SecurityFilterChain` configuration to Security 7 idioms (lambda DSL strictness, current authorization API). This is in-scope, not deferred.
- **Test harness**: sweep all tests off deprecated/removed Boot test APIs in one pass (e.g., `@MockBean` → `@MockitoBean`, MockMvc/WebTestClient API changes). Cleaner break than fixing case-by-case.
- **Property keys**: enable `spring-boot-properties-migrator` during the upgrade, surface every renamed key, and migrate all of them. Then remove the migrator before merge.

### Definition of done
- **Closure gate**: full unit + integration test suite green, plus a manual end-to-end smoke of the existing Nango integration sync path (PROJECT constraint: `IntegrationSyncWorkflowImpl` must keep working unchanged).
- **Required smoke coverage**: the Nango integration sync path. Other zones (auth, Temporal workflows, JPA + pgvector) are covered by the test suite — surface them only if the test suite gap is found.
- **Rollback**: branch-only. The upgrade ships on a feature branch; if blocked, abandon the branch and main stays on 3.5.3. No flag-gated half-upgrade.
- **No perf baseline**: don't gate on startup time / memory. If an obvious regression surfaces in dev, raise it; otherwise ship.

### Claude's Discretion
- Whether to introduce a Spring Boot BOM `dependencyManagement` block vs continuing to use the `io.spring.dependency-management` Gradle plugin.
- Exact ordering of dep bumps inside the phase (e.g., bump Hibernate first, then Boot, vs all at once).
- Per-test migration mechanics for the test sweep.
- Specific property key migrations beyond what `spring-boot-properties-migrator` flags.
- Whether to bump `io.spring.dependency-management` plugin version itself.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `core/build.gradle.kts`: single source of truth for backend deps. All version changes happen here.
- Existing `SecurityFilterChain` config (Spring Security): the migration target for Security 7 idioms.
- Existing `IntegrationSyncWorkflowImpl` + Temporal worker config: smoke-test target post-upgrade.
- `application.yml` / Spring profiles: target for `spring-boot-properties-migrator` sweep.

### Established Patterns
- Gradle Kotlin DSL with `io.spring.dependency-management:1.1.7` + `org.springframework.boot:3.5.3` plugins managing versions for most Spring artifacts.
- Some Spring Security artifacts are pinned to explicit versions (3.5.0 / 6.5.0) — these need to be unpinned to let the Boot 4 BOM take over.
- Java toolchain set explicitly to 21 in `java { toolchain { ... } }` — stays.
- Tests use Spring Boot test starter + Mockito + Testcontainers (Postgres). H2 also present.

### Integration Points
- `core/build.gradle.kts` plugins block: `org.springframework.boot` plugin version is the upgrade trigger.
- BOMs in play: Supabase BOM (3.1.4) — verify it doesn't fight Boot 4's Jackson/Ktor versions.
- Hypersistence + hibernate-vector: artifact name changes when bumping major version (they encode Hibernate major in the artifact ID).

</code_context>

<specifics>
## Specific Ideas

- The whole point of this upgrade is to unblock Phase 03-04 (Spring AI 2.x for multi-provider NL-assisted mapping with Anthropic default + OpenAI reasoning). Spring AI 2.x is the next-phase consumer; do not pull it in here.
- The Nango integration sync path is the canary. If it breaks, the upgrade isn't done.
- Prefer abandoning the branch over merging a partial/shimmed Boot 4 upgrade.

</specifics>

<deferred>
## Deferred Ideas

- **Java 25 / Kotlin 2.2 bumps** — separate phase, after Boot 4 is stable.
- **Opportunistic non-Spring dep refresh** (Tika, PostHog, Jackson, etc.) — separate hygiene phase.
- **Non-critical deprecation cleanup sweep** — tech-debt phase after Boot 4 lands.
- **Startup time / memory perf baseline + regression tracking** — observability initiative, not this phase.
- **Spring AI 2.x integration itself (MAP-07 / NL-assisted mapping)** — Phase 03-04, reopens after this phase closes.

</deferred>
